什么是Redis？简述它的优缺点

redis本质上是一个key-value类型的内存数据库，把数据加载到内存，操作在内存中进行

优点：操作在内存进行，所以处理速度非常快，而且支持保存多种数据结构

缺点：容易受到内存限制，不能用于海量数据的高性能读写

Redis有哪些数据结构(类型)

博客链接：redis的五种数据结构

redis有五种基础数据结构

  • string ：普通字符串，常用做缓存
  • list ：双端链表 实现最新消息排队功能。根据命令方法可以实现栈，队列，有限集合，消息队列
  • hash： string类型的字段和属性的映射表，适合存储对象
  • set： string类型的无序集合，没有重复元素，集合通过hash表实现，常用作点赞点踩等功能
  • zset ：每个元素都会关联一个double类型的权重参数，按照这个权重来进行排序的。有序集合，没有重复元素，适合排行榜

redis是单线程还是多线程，为什么这么快

  1. 完全基于内存操作。它的数据存储在内存中，这样能够获得非常快的读写速度
  2. 降低CPU的消耗，单线程减少不必要的上下文切换，不用考虑各种锁的问题

  3. Redis提供了多种高效的数据结构，如字符串、哈希表、列表、集合、有序集合等。对实现经过了高度优化

    • 整数编码：会根据整数的大小选择最节省内存的编码方式来存储数据
    • 压缩列表：针对列表类型数据，可以最小化存储空间，快速访问数据
    • 哈希表：针对哈希表数据类型，实现高效插入、删除和查找操作
    • 跳跃表：针对有序集合数据类型，采用跳跃表，实现高效地插入、删除和查找操作

  4. 采用更高效的非阻塞异步IO，采用epoll作为io多路复用技术的实现。

redis在6.0版本之前是单线程，在这之后是多线程。在单线程下省去了线程切换，和锁竞争的开销，比较快

更重要的是它基于内存操作

redis如何实现分布式锁

为什么redis要将所有数据放到内存中

是为了达到最快的读写速度而放在内存中，并且通过异步的方式写入到磁盘中

如果不放在内存中，磁盘io速度会影响redis性能

缓存失效、缓存穿透、缓存雪崩、缓存并发

博客链接：详解缓存穿透、缓存雪崩、缓存击穿

【**缓存雪崩**】

  • 定义：大量热点key设置相同的过期时间，导致缓存在同一时间全部失效，导致瞬时数据库请求量大
  • 解决办法：
    • 给缓存的过期时间加上一个随机值时间，将过期时间分散开
    • 热点数据考虑不失效，后台异步更新缓存，适用于不严格要求缓存一致性的场景
    • 考虑队列或者锁的方式，可能影响并发量

【**缓存穿透**】

  • 定义：查询一个数据库不存在的数据，由于缓存的机制，每次查询都会从数据库找数据，然而数据库没有，此时大量的请求就直接打到数据库
  • 解决办法
    • 可以在封装get和set时增加一个步骤，如果查询的key不存在，就以这个key为前缀设定一个表示key，等下一次查询该key时，先查询标识key，如果该key存在，就返回协定好的非false或者null值
    • 当返回结果是空时，仍然把这个空结果进行缓存里面
    • 采用布隆过滤器请求白名单。过滤器先判断是否再白名单内，不在就直接返回空，或失败
    • 接口层增加校验，加强用户权限校验，提前阻断非法请求

【**缓存击穿**】

  • 定义：单个热点key，在缓存过期的一刻，大量请求会打入数据库，造成压力
  • 解决办法：
    • 使用互斥锁，只让一个线程创建缓存，其他线程等待缓存创建完毕之后，再从缓存中获取数据。保证一致性，但是性能受影响，有死锁风险
    • 热点数据不设置过期时间，后台异步更新缓存。不保证一致性，有内存消耗

缓存失效

  • 定义：是指一个缓存在高并发下，当一个缓存在某一时间失效时，在高并发下，导致数据库的瞬间压力过大
  • 解决办法：可以将缓存的失效时间分散开

缓存并发

  • 定义：高并发下，如果一个缓存失效，需要同时设置缓存，如果并发量过高，也能导致数据库压力过大
  • 解决办法：对缓存加锁，如果不存在，就加锁，然后查询数据库加入缓存，然后解锁。其他进程如果发现有锁时，就等待其解锁

Redis中的热key怎么处理

博客链接：redis中热点key问题

缓存中的某些key对应的value存储在集群中的一台机器上，使得所有流量涌向同一台机器，限制性能

  • 使用二级缓存。将热点key对应的value存储在客户端本地，并设置一个失效时间

  • 将热点key分散为多个子key，存储在不同机器上。当通过热点key去查询数据时，通过某种hash算法随机选择一个子key，然后再访问对应缓存机器

Redis的缓存淘汰策略

当redis使用内存达到maxMemory时，redis会根据配置的缓存淘汰策略，把访问频率不高的key，从内存中移除掉

  • 采用LRU淘汰算法策略。把不常用的key淘汰掉
  • 采用LFU策略。对LRU进行优化，增加数据访问频次，确保淘汰的数据是非热点的，最小访问频率的
  • 随机策略。随机删除key
  • ttl策略。从设置了过期时间key里面，挑选过期时间最近的key进行淘汰
  • 直接报错。当内存不够时直接抛出异常

直接报错，内存不够直接抛出异常

使用淘汰算法，淘汰掉不常用的key

使用LFU策略，是淘汰算法的优化，增加了一个数据访问频次，当内存满了时，优先淘汰掉数据频次小的

随机策略，随机删除key

使用ttl策略，从设置的过期时间的key，使用对应的LRU或者LFU策略进行删除淘汰key
